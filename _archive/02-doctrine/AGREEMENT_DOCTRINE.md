# Agreement Is All We Need

## The OMI Doctrine of Bounded, Replayable Trust

```text
omi---imo
o---o
```

Agreement is the moment an open-world declaration becomes bounded, validated, replayable, and inspectable.

It is not a feeling, a vote, a platform decision, or the fact that something rendered on a screen.

In OMI, agreement is a structural event:

```text
A declaration enters the frame.
A transition resolves.
A receipt records.
A materializer applies.
A projection displays.
An inspector explains.
```

The spine is:

```text
validate -> resolve -> record -> materialize -> project -> inspect
```

The forbidden shortcut is:

```text
render -> believe
```

OMI exists to prevent that shortcut.

## The Problem

Most digital systems confuse visibility with truth.

If a page renders, people believe it exists. If a platform shows a badge, people believe it is trusted. If a database returns a value, people believe it is current. If a social feed repeats a claim, people believe it has weight.

This is the old mistake:

```text
render -> believe
```

OMI rejects this. Rendering is only projection. Projection can help a person read an object, but projection cannot validate the object.

A DOM node can carry state. A CSS rule can style state. A JSON object can serialize state. A QR code can transport state. A gossip network can recover state. A portal can display state.

None of these are authority by themselves.

The first doctrine is:

```text
carrier != authority
projection != validation
prefix != identity
gossip != authority
render != believe
```

## What Agreement Means

In everyday language, agreement often means:


In OMI, agreement means something stricter:

```text
the object has a boundary

the receipt records
the history can replay
the projection can be inspected
```

Agreement is not the absence of disagreement. Agreement is the production of enough structure that disagreement can be located, replayed, compared, and resolved.

A disagreement says:

```text

```

OMI asks:

```text
What boundary is missing?
What source is missing?
What receipt is missing?
What context is missing?
What transition failed?
What projection is being mistaken for authority?
```

Then it turns the disagreement into an object path.

```text
disagreement -> missing structure -> new object/path/test/receipt
```

## The 128-Bit Agreement Frame

An OMI frame has eight 16-bit segments:

```text
S0-S1-S2-S3-S4-S5-S6-S7
```

Together:

```text
8 x 16 = 128 bits
```

The canonical frame shape is:

```text
S0 = 0xLL00
S1 = 0x03BF
S2 = 0xNNNN
S3 = 0x2BLL
S4 = 0x2FLL
S5 = 0xMMMM
S6 = 0x039F
S7 = 0xLLFF
```

The frame contains:

```text
LL = scope / lens / ket selector
NN = antecedent
MM = consequent
```

The frame validates only if:

```text
Q_frame(S) = 0
```

The structure must close. The object cannot become accepted by looking correct. It must satisfy the frame.

## The Truth Row

Every valid frame yields one truth row:

```text
(LL, NN, MM)
```

This is the smallest agreement face. It can be read as:

```text
scope / antecedent / consequent
lens / beginning / resolution
declaration / definition / receipt
```

The proven transition is:

```text
NN -> MM
```

`NN` begins. `MM` resolves. `LL` scopes where the accepted transition lands.

The transition is accepted only when Delta/Fano proves that `MM` is reachable from `NN` inside the bounded path.

```text
LL scopes.
NN begins.
MM resolves.
```

## Delta Proves, Fano Bounds

The Delta Law is the bitwise transition engine.

It uses bounded operations:

```text
rotate
xor
mask
constant
```

A simple form is:

```text
Delta_C(x) = rotl(x,1) xor rotl(x,3) xor rotr(x,2) xor C
```

The resolver attempts:

```text
NN -> Delta(NN) -> ... -> MM
```

The Fano layer bounds the incidence search. It asks which triangle resolves, which relation closes, and which path can be replayed.

The path is finite. The proof is local. The acceptance is receipted.

```text
Delta proves.
Fano bounds.
Receipt records.
```

## LL as the Ket of Agreement

`LL` is the ket-like selector of the frame.

It is not the object, geometry, or policy. It is the scope selector.

The truth row can be read:

```text
(|LL>, NN, MM)
```

`LL` has two nibbles:

```text
LL = 0xAB
```

Where:

```text
A = high nibble
B = low nibble
```

OMI reads them as:

```text
high nibble = context / lens family / carry-forward scope
low nibble  = execution slot selector
```

The low nibble selects where the accepted transition materializes. The high nibble carries scope forward.

The gate extracts:

```sh
slot_index=$((ll & 0x0f))
rule_slot=$((0xe0 + slot_index))
context=$(((ll >> 4) & 0x0f))
```

The shell does not know meaning. It only emits the accepted envelope.

## The 0xE_ Agreement Manifold

The `0xE_` band is the accepted execution manifold:

```text
0xE0-0xEF
```

The low nibble of `LL` selects one of sixteen slots:

```text
rule_slot = 0xE0 + (LL & 0x0F)
```

The first twelve slots pack the object:

```text
0xE0-0xEB
```

The final four slots hold the object:

```text
0xEC-0xEF
```

The doctrine is:

```text
0xE0-0xEB pack the object.
0xEC-0xEF hold the object.
```

The first twelve are the inner materializer field. The final four are the outer controller vertices. Together they form one accepted execution manifold.

```text
No controller exists outside receipt.
```

## Outer Controller Vertices

The final four slots control the projection envelope:

```text
0xEC = translation / viewport / focus
0xED = scale / recursion depth
0xEE = rotation / spin / orientation
```

These are not UI shortcuts. They are accepted controller addresses.

The rule is:

```text

```

A local browser may scroll, pan, rotate, or zoom as local UI. But a conforming OMI projection must not treat that local behavior as accepted state.

The viewport is receipt-bound when controlled through `0xEC-0xEF`. Out-of-band lens changes are local display only.

## Agreement and Meta-Memory

Agreement operates on five meta-memory families. Each family has a distinct role in the agreement lifecycle:

| Family | Keyword | Role | Canon |
|---|---|---|---|
| RULES | MUST | normative invariants | RULES constrain |
| FACTS | FACT | ground truth | FACTS ground |
| CLOSURES | CLOSE | bounded completion | CLOSURES seal |
| COMBINATORS | COMBINE | lawful transformation | COMBINATORS compose |
| CONS | CONS | pair structure | CONS reduce |

### RULES

A rule is a normative obligation: `<address>/<prefix> MUST <invariant>`. Rules declare what must be true for conformance. They do not claim implementation.

### FACTS

A fact grounds a rule: `<address>/<prefix> FACT <name>`. A fact says a rule has been implemented, tested, and projected. Facts are derived from rules through the rule-to-fact derivation protocol.

### CLOSURES

A closure seals bounded completion: `<address>/<prefix> CLOSE <name>`. Closures enable WLOG-safe reasoning — without a closure, a rewrite is merely local and cannot generalize.

### COMBINATORS

A combinator declares lawful transformation: `<address>/<prefix> COMBINE <name>`. Combinators compose operations without validating frames by themselves.

### CONS

A cons declares pair structure: `<address>/<prefix> CONS <name>`. Cons records define pairing, nesting, dot notation, and mnemonic forms.

### The Four Operators

Four channel operators traverse the meta-memory families:

| Operator | Motion | Effect |
|---|---|---|
| JOIN | binds | merges families into composite structure |
| COMPOSE | orders | sequences families by dependency |
| ENCODE | lowers | reduces abstract to concrete |
| DECODE | raises | recovers abstract from concrete |

The operators act on every family:

```text
JOIN rules, facts        → grounded obligation
COMPOSE rules, facts     → derivation order (rules before facts)
ENCODE rule → fact       → rule grounded in implementation
DECODE fact → rule       → recover invariant from ground truth
```

### Twelve Tribes of Agreement

The Twelve Tribes group into three quads:

| Quad | Tribe | Role |
|---|---|---|
| Foundation | Freedom | open declaration |
| Foundation | Autonomy | self-directed frame |
| Foundation | Sovereignty | bounded identity |
| Foundation | Reciprocity | mutual validation |
| Attention | Focus | object selection |

| Attention | Observation | evidence inspection |
| Attention | Experience | interaction memory |
| Boundary | Stop | rejection boundary |
| Boundary | Yes | acceptance signal |
| Boundary | No | refusal signal |
| Boundary | Acknowledgement | receipt acknowledgment |

The Foundation tribes establish preconditions for agreement. The Attention tribes govern how users encounter agreement surfaces. The Boundary tribes mark the edge of agreement.
